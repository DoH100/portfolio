Search Performance
검색 속도 13배 향상 (Elasticsearch)
[문제상황] MySQL LIKE 검색(690ms)은 Full Table Scan을 유발했고, 형태소 분석이 불가능했습니다. Full-Text Search(302ms) 또한 확장성 및 한글 처리에 한계가 있었습니다.
[해결방법] Elasticsearch를 도입하여 평균 응답 시간을 690ms → 52ms로 약 13배 단축시키고, 정확도 높은 검색 환경을 구축했습니다.
Database Stability
DB 부하 800% → 30% 안정화
[문제상황] 단일 DB 인스턴스에 읽기/쓰기 트래픽이 모두 집중되어, 피크 시간대 CPU 부하가 800%까지 급등하는 장애 위험이 있었습니다.
[해결방법] Master/Slave Replication 아키텍처를 구축하여 읽기 트래픽을 Slave로 분산시켰습니다. 그 결과 Master DB의 CPU 부하를 30% 수준으로 안정화했습니다.
High Traffic
입찰 TPS 1.5배 증가
[문제상황] 실시간 입찰 시 조회/갱신 작업이 DB에 집중되어 CPU 피크(9% 이상)를 유발하고 전체적인 처리량(TPS)이 저하되었습니다.
[해결방법] 고빈도 입찰 데이터를 Redis(In-memory)에서 우선 처리하도록 분산했습니다. 입찰 처리량(TPS)이 1.5배 증가하고 CPU 피크 사용률은 50% 절감되었습니다.
Query Tuning
쿼리 속도 70% 단축
[문제상황] 경매 상태 및 입찰자 등 복합 조건 조회 시 Full Table Scan이 발생하여 평균 쿼리 응답 시간이 417.6ms로 지연되었습니다.
[해결방법] 조회 패턴을 분석하여 `last_bidder_id`, `status`, `end_at` 순서의 복합 키 인덱스를 적용, 응답 시간을 124.0ms로 단축했습니다.

스레드 경합으로 인한 지연(19.6초) 해결. 블로킹 I/O 처리에 효율적인 Virtual Thread를 적용하여 TPS 2.7배 증가 및 에러율 0% 달성.
